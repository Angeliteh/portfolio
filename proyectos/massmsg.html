Por qué banean a los bots estándar

Las herramientas cloud de automatización inyectan requests o usan navegadores headless desde IPs de datacenter. Las plataformas lo detectan de inmediato. MassMSG controla tu Chrome local real, escribe letra por letra con delay variable y hasta corrige errores como una persona.

El problema

Las APIs oficiales de mensajería son caras, restrictivas y detectables.

WhatsApp Business API cobra por conversación y exige aprobación previa de plantillas. Las herramientas cloud de envío masivo usan IPs compartidas de datacenter que son detectadas y bloqueadas en minutos. Muchas opciones open source además requieren alto conocimiento técnico.

Las PyMEs necesitaban una solución local que corriera en su propia máquina, usara su cuenta real y su IP residencial, y se viera completamente humana para los sistemas de seguridad de la plataforma.

Arquitectura: diseño de doble interfaz

Un motor, dos interfaces: local, portable y sin dependencia cloud.

Electron Main  Orchestrates lifecycle
    └─ Spawns FastAPI Server (Python binary)
    └─ Renders React UI via Vite (dev) / dist/ (prod)
    └─ Graceful shutdown via POST /api/shutdown + SIGKILL fallback

FastAPI Backend  Local API on :8000
    └─ /launch     → Finds Chrome, opens debug port :9222
    └─ /init       → Attaches Playwright via CDP
    └─ /send-batch → Queues message campaigns
    └─ /api/campaigns → Campaign CRUD (JSON persistence)

Playwright Core  Attached to native Chrome via CDP
    └─ Manipulates WhatsApp / Gmail / Google Messages DOM
    └─ Typo Engine → human-realistic keystrokes
    └─ Bio-pauses → randomized delays, "lunch breaks"

Heartbeat System  Process integrity
    └─ React pings /api/heartbeat every 5s
    └─ If >15s without ping → Python self-terminates (os._exit)
    └─ Prevents zombie processes on Electron crash
Motor stealth
Typo Engine: emulación humana de tecleo

Delays variables por tecla, errores aleatorios corregidos con Backspace y pausas de pensamiento simuladas. Los caracteres se escriben uno por uno, nunca se inyectan.

Bio-pauses: simulación de jornada real

Pausas aleatorias por distracción, descansos de café y breaks largos en lotes grandes. Cada delay tiene variación aleatoria para evitar patrones detectables.

IP residencial y fingerprint nativo

Corre sobre el Chrome real del usuario con su fingerprint verdadero e IP doméstica, idéntico a una operación manual. Las plataformas no pueden distinguirlo de uso orgánico.

Sesión persistente y portabilidad

El perfil de Chrome se guarda en prod_browser_profile/ y el QR se escanea solo una vez. Mover la carpeta a otra unidad conserva la sesión completa sin reconfiguración.

Seguridad: licenciamiento RSA + HWID

Protección criptográfica en tres capas contra piratería.

Layer 1 · Startup
    └─ Reads license.key + generates HWID from Motherboard UUID
    └─ Verifies RSA signature with embedded public key
    └─ 7-day offline grace period via encrypted local cache

Layer 2 · Middleware
    └─ FastAPI middleware intercepts every HTTP request
    └─ Returns HTTP 403 if license_manager.is_active() = false

Layer 3 · Action Gate
    └─ RSA signature re-validated immediately before send_message()
    └─ Prevents in-memory manipulation of license state

Desafíos técnicos y soluciones

Bloqueos por APIs y herramientas cloud

Problema

Las herramientas SaaS cloud de mensajería masiva son bloqueadas rápidamente porque Meta detecta sus IPs de datacenter y firmas de navegador headless.

Solución

MassMSG es una app local de escritorio. Aprovecha la IP residencial orgánica del usuario y el fingerprint nativo del navegador, volviéndolo indistinguible del uso humano.

Procesos zombie huérfanos

Problema

Al empaquetar Python con Electron, si el usuario cierra a la fuerza la UI, el backend Python puede seguir vivo en RAM como fuga de memoria.

Solución

Sistema Heartbeat de doble seguridad. La UI React envía ping cada 5s y, si Python detecta 15s sin señal, se auto-termina de inmediato para evitar procesos colgados.
