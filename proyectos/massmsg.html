Why standard bots get banned

Cloud automation tools inject API requests or use headless browsers from datacenter IPs. Platforms detect this instantly. MassMSG takes control of your local, normal Chrome profile. It types letters one by one using a variable delay. If it makes a mistake, it presses Backspace. It literally emulates a human worker.

The Problem

Official messaging APIs are expensive, restrictive, and detectable.

WhatsApp Business API charges per conversation and requires template pre-approval. Cloud-based bulk messaging tools use shared datacenter IPs that get detected and banned within minutes. Existing open-source RPA tools require deep technical knowledge to configure and maintain.

SMBs needed a solution that runs locally on their own machine, uses their actual account and residential IP, and looks completely human to the platform's security systems.

Architecture: Dual Interface Design

One engine, two interfaces — local, portable, zero cloud dependency.

Electron Main  Orchestrates lifecycle
    └─ Spawns FastAPI Server (Python binary)
    └─ Renders React UI via Vite (dev) / dist/ (prod)
    └─ Graceful shutdown via POST /api/shutdown + SIGKILL fallback

FastAPI Backend  Local API on :8000
    └─ /launch     → Finds Chrome, opens debug port :9222
    └─ /init       → Attaches Playwright via CDP
    └─ /send-batch → Queues message campaigns
    └─ /api/campaigns → Campaign CRUD (JSON persistence)

Playwright Core  Attached to native Chrome via CDP
    └─ Manipulates WhatsApp / Gmail / Google Messages DOM
    └─ Typo Engine → human-realistic keystrokes
    └─ Bio-pauses → randomized delays, "lunch breaks"

Heartbeat System  Process integrity
    └─ React pings /api/heartbeat every 5s
    └─ If >15s without ping → Python self-terminates (os._exit)
    └─ Prevents zombie processes on Electron crash
Stealth Engine
Typo Engine — Human Keystroke Emulation

Variable per-key delays (0.10–0.20s), random "fat finger" typos with Backspace correction, and simulated thinking pauses every N characters. Characters are typed one by one — never injected.

Bio-Pauses — Office Day Simulation

Randomized distraction pauses (10–30s), coffee breaks (10–20 min every 50 messages), and lunch breaks (30–60 min on large batches). Every delay has ±25% random variance — no detectable pattern.

Residential IP + Native Fingerprint

Runs on the user's Chrome with their actual browser fingerprint and home IP — identical to a human navigating manually. Platforms cannot distinguish it from organic user activity.

Persistent Session & Portability

Chrome profile stored in prod_browser_profile/ — one-time QR scan. Moving the folder to an external drive preserves the full session with zero reconfiguration.

Security: RSA + HWID Licensing

Three-layer cryptographic protection against piracy.

Layer 1 · Startup
    └─ Reads license.key + generates HWID from Motherboard UUID
    └─ Verifies RSA signature with embedded public key
    └─ 7-day offline grace period via encrypted local cache

Layer 2 · Middleware
    └─ FastAPI middleware intercepts every HTTP request
    └─ Returns HTTP 403 if license_manager.is_active() = false

Layer 3 · Action Gate
    └─ RSA signature re-validated immediately before send_message()
    └─ Prevents in-memory manipulation of license state

Technical Challenges & Solutions

Cloud API Bans

Problem

Cloud-based mass messaging SaaS tools get banned instantly because Meta detects their datacenter IP addresses and headless browser signatures.

Solution

MassMSG is a local desktop app. It leverages the user's organic residential IP and native browser Canvas fingerprint, making it indistinguishable from human use.

Orphaned Zombie Processes

Problem

When bundling Python with Electron, if the user force-quits the UI, the Python backend stays alive in the RAM forever as a memory leak.

Solution

A double-safety Heartbeat system. The React UI sends a ping every 5s. If Python detects 15s without a ping, it self-terminates instantly (os._exit), keeping the user's PC clean.
